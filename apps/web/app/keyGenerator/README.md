这个工具用于在当前浏览器页面中即时生成常见开发密钥，支持以下首版类型：

1. `API Key`
2. `JWT Secret`
3. `AES-256 Key`
4. `HMAC-SHA256 Secret`
5. `RSA Key Pair`

## 支持的输出格式

- `API Key`: 纯文本、Hex、Base64、Base64URL
- `JWT Secret`: Hex、Base64、Base64URL
- `AES-256`: Hex、Base64、Base64URL、JWK
- `HMAC-SHA256`: Hex、Base64、Base64URL、JWK
- `RSA Key Pair`: PEM、JWK

## 密钥类型说明

### `API Key`

`API Key` 通常用于标识调用方身份，常见于开放平台、内部系统对接、测试环境访问控制等场景。它本质上更像一个随机凭证，而不是一个有明确密码学算法约束的密钥。

- 适用场景：开放 API 调用、内部接口鉴权、临时访问令牌、系统集成测试。
- 特点：实现简单、接入成本低、常直接放在请求头或查询参数中。
- 注意点：`API Key` 一旦泄露，通常就意味着调用权限泄露，因此仍然需要妥善保管。

### `JWT Secret`

`JWT Secret` 一般用于基于共享密钥的 JWT 签名算法，例如 `HS256`、`HS384`、`HS512`。签发方和验签方使用的是同一份密钥，因此它属于对称密钥。

- 适用场景：服务端签发登录态 Token、内部服务间 JWT 鉴权、临时签名验证。
- 特点：配置简单，适合单体应用或受控的服务体系。
- 注意点：签发和验证共用同一密钥，泄露风险高于公私钥分离方案。

### `AES-256`

`AES-256` 是一种常见的对称加密密钥，表示使用 256 位长度的 AES 密钥。生成出来的是原始加密密钥，可用于 `AES-GCM` 等模式的加密和解密。

- 适用场景：本地数据加密、配置加密、敏感字段加密、前后端协同加密实验。
- 特点：加密与解密使用同一把密钥，性能好，适合大多数对称加密需求。
- 注意点：密钥分发和保存是核心风险点，只要拿到密钥就能解密数据。

### `HMAC-SHA256`

`HMAC-SHA256` 不是用来“加密数据”的，而是用来生成消息摘要签名，验证一段内容在传输中是否被篡改，以及内容是否来自可信的一方。

- 适用场景：Webhook 签名、回调验签、请求防篡改、服务间消息完整性校验。
- 特点：速度快，实现广泛，适合签名和验签而不是数据加密。
- 注意点：它只能证明完整性和来源，不能直接用于恢复明文，也不能替代加密算法。

### `RSA Key Pair`

`RSA Key Pair` 是经典的非对称密钥方案，由公钥和私钥组成。公钥可分发给他人，私钥只应由持有者自己保管。当前工具生成的 RSA 密钥对适合加密、解密、身份校验等开发测试场景。

- 适用场景：公私钥分发、服务端加密解密、系统集成调试、密钥交换实验。
- 特点：公钥和私钥职责分离，适合多方协作和更清晰的权限边界。
- 注意点：私钥必须严格保密；公钥泄露通常没问题，但私钥泄露会直接失去安全性。

## 编码格式区别

### `hex`

`hex` 是十六进制文本表示方式，每个字节会被编码成两个字符，例如 `ff`、`0a`。

- 优点：可读性强，排查问题时很直观。
- 缺点：体积偏大，通常会比原始二进制数据大一倍。
- 适合：日志观察、手动比对、某些只接受十六进制字符串的系统。

### `base64`

`base64` 会把二进制数据编码成更紧凑的文本格式，常见字符包括 `A-Z`、`a-z`、`0-9`、`+`、`/`，有时末尾会带 `=` 补位。

- 优点：比 `hex` 更短，更适合网络传输和配置存储。
- 缺点：包含 `+`、`/`、`=`，在 URL、文件名或部分系统中不够友好。
- 适合：配置文件、环境变量、通用数据交换。

### `base64url`

`base64url` 是 `base64` 的 URL 安全变体，通常会把 `+` 替换为 `-`，把 `/` 替换为 `_`，并去掉尾部的 `=`。

- 优点：适合 URL、Cookie、JWT 等对字符集有限制的场景。
- 缺点：有些系统只接受标准 `base64`，接入前需要确认格式要求。
- 适合：JWT、URL 参数、浏览器环境中的轻量文本传输。

### `PEM`

`PEM` 是一种常见的文本封装格式，通常用于表示公钥、私钥、证书等内容。它本质上一般是把二进制密钥数据做 `base64` 编码后，再加上类似 `-----BEGIN PUBLIC KEY-----` 的头尾标记。

- 优点：兼容范围广，很多后端框架、OpenSSL 工具链和基础设施系统都支持。
- 缺点：主要适用于非对称密钥或证书，不适合普通随机字符串密钥。
- 适合：RSA 公钥、RSA 私钥、证书链相关的导入导出。

### `JWK`

`JWK` 全称是 `JSON Web Key`，是一种用 JSON 结构表示密钥的格式。它不仅保存密钥材料，还可能包含算法、用途、曲线、模数等结构化信息。

- 优点：结构化强，前后端处理方便，和 Web 标准生态配合很好。
- 缺点：对不熟悉 JSON Key 规范的系统来说，可读性不如 `PEM` 直观。
- 适合：Web Crypto API、现代前端应用、JWT/JOSE 生态、浏览器侧密钥交换。

## `Web Crypto API` 的作用

`Web Crypto API` 是浏览器内置的密码学能力接口，允许前端在本地安全地生成随机数、创建密钥、导出密钥，并执行签名、验签、加密、解密等操作。

在这个工具中，它主要承担以下职责：

1. 通过 `crypto.getRandomValues()` 生成高质量随机字节，用于 `API Key`、`JWT Secret` 等随机内容。
2. 通过 `crypto.subtle.generateKey()` 生成 `AES-256`、`HMAC-SHA256` 和 `RSA Key Pair`。
3. 通过 `crypto.subtle.exportKey()` 将密钥导出为 `raw`、`JWK`、`SPKI`、`PKCS8` 等格式，再进一步转换成页面展示的 `hex`、`base64`、`base64url` 或 `PEM`。

它的核心价值在于：生成过程发生在用户当前浏览器环境中，不需要把密钥发送到服务端，从而更符合这个工具“不上传服务器、不落地本地存储”的安全设计。

## 使用说明

1. 在左侧选择密钥类型、输出格式和对应参数。
2. 点击“生成”后，右侧展示当前结果。
3. 敏感结果默认隐藏，可手动切换显示状态。
4. 你可以按需复制或下载某一段结果。
5. 点击“恢复默认配置”会回到初始页面设置。

## 安全说明

本工具仅在当前浏览器页面内存中生成和展示密钥内容。

1. 不会将生成结果上传到服务器。
2. 不会将生成结果写入本地存储。
3. 不会保存历史记录、草稿或最近一次生成结果。
4. 每次进入页面都会使用默认配置。
5. 刷新或关闭页面后，当前配置和生成结果都会被清空。

如果你主动执行复制或下载，密钥内容将由浏览器、操作系统或本地文件系统接管，应用本身不会额外留存。

## 注意事项

- 本工具依赖浏览器提供的 `Web Crypto API`。
- 不同系统对 `PEM`、`JWK` 或 `Base64URL` 的接入方式可能不同，请按目标系统要求使用。
- 生成私钥后请及时保存到安全位置，避免刷新页面后丢失。

## 常见问题

### `JWT Secret` 和 `API Key` 有什么区别

- `JWT Secret` 更适合做 `HS256`、`HS384`、`HS512` 这类签名算法的共享密钥
- `API Key` 更像访问凭证或调用标识，不等同于密码学意义上的签名密钥

### 什么时候应该选 `RSA Key Pair`

当你的系统需要公私钥分离、签发方和验签方职责分离、或需要把公钥分发给第三方时，更适合使用 `RSA Key Pair`。

### `Base64` 和 `Base64URL` 应该怎么选

- 需要放进 URL、Cookie、JWT 或文件名时，更适合 `Base64URL`
- 目标系统只接受传统编码时，优先使用标准 `Base64`
