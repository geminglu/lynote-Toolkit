这个工具用于在当前浏览器页面中完成常见 `RSA` 联调操作，适合前后端协作、第三方平台接入、自测签名验签和密钥格式排查。

## 首版支持

### 操作模式

1. `加密`
2. `解密`
3. `签名`
4. `验签`
5. `密钥检查`

### 密钥格式

1. `PEM`
2. `JWK`

### 加密能力

- 算法：`RSA-OAEP`
- Hash：`SHA-256`、`SHA-384`、`SHA-512`、`SHA-1`

### 签名能力

- 算法：`RSASSA-PKCS1-v1_5`
- 算法：`RSA-PSS`
- Hash：`SHA-256`、`SHA-384`、`SHA-512`、`SHA-1`

### 二进制编码

1. `Base64`
2. `Base64URL`
3. `Hex`

## RSA 的常见用途

### `RSA-OAEP`

适合用公钥加密短文本，例如：

- 一次性对称密钥
- 小段敏感参数
- 联调用测试明文

### `RSASSA-PKCS1-v1_5`

适合传统系统或旧平台的签名验签流程，兼容性较好。

### `RSA-PSS`

更现代的签名方案，通常比传统 `PKCS1 v1.5` 更稳健，但需要签名和验签双方明确约定 `saltLength`。

## 加密与签名的区别

### `加密 / 解密`

- 公钥加密
- 私钥解密
- 目标是保护明文内容，恢复原文

### `签名 / 验签`

- 私钥签名
- 公钥验签
- 目标是证明来源和完整性，不用于恢复原文

## PEM 和 JWK 的区别

### `PEM`

- 更适合人工复制、OpenSSL、后端框架和基础设施场景
- 常见头尾格式如 `-----BEGIN PUBLIC KEY-----`

### `JWK`

- JSON 结构化表示方式
- 更适合浏览器、JOSE、JWT 和现代 Web 生态

## 编码格式区别

### `Base64`

- 最常见
- 传输紧凑
- 可能包含 `+`、`/`、`=`

### `Base64URL`

- URL 更友好
- 常见于 JWT / JOSE 生态

### `Hex`

- 可读性强
- 更适合日志和人工比对
- 文本体积更大

## 密钥检查能做什么

在“密钥检查”模式下，工具会尽量识别：

- 当前是公钥还是私钥
- 当前是 `PEM` 还是 `JWK`
- 当前 RSA 位数
- 当前浏览器成功识别到的算法结构
- 当前密钥在该识别方式下的可用用途

这适合在联调时快速确认“我拿到的到底是什么密钥”。

## 为什么 RSA 不适合长文本

`RSA` 更适合处理短数据，而不是直接加密整段长文本或文件。

原因是：

- 可加密内容长度受密钥位数和摘要算法影响
- 明文一旦超过上限就会失败
- 更常见的真实方案是 `RSA + AES` 混合加密

因此，这个工具会在加密时提示当前密钥下的最大明文长度。

## 使用说明

1. 在左侧选择操作模式。
2. 选择密钥格式和算法参数。
3. 粘贴公钥或私钥。
4. 按模式输入明文、密文、原文或签名值。
5. 点击“执行”。
6. 在右侧查看密钥摘要、结果和兼容性提示。
7. 你可以复制或下载任一结果。
8. 点击“恢复默认配置”回到初始状态。

## 安全说明

本工具仅在当前浏览器页面内存中处理 `RSA` 密钥、明文、密文与签名值。

1. 不会将输入内容上传到服务器。
2. 不会自动写入本地存储。
3. 不会保存历史记录。
4. 刷新或关闭页面后，当前内容会被清空。
5. 如果你主动执行复制或下载，数据将由浏览器和操作系统接管。

## 限制说明

- 当前加密模式仅支持浏览器 `Web Crypto API` 可稳定支持的 `RSA-OAEP`
- 当前私钥 PEM 优先支持 `PKCS8`，同时兼容常见 `RSA PRIVATE KEY`
- 当前公钥 PEM 优先支持 `SPKI`，同时兼容常见 `RSA PUBLIC KEY`
- `SHA-1` 仅建议用于兼容旧系统
- 解密结果默认按 `UTF-8` 文本展示，不适合直接展示任意二进制明文

## 代码示例

### Node.js 中使用公钥加密

```ts
import { publicEncrypt } from "node:crypto";

const ciphertext = publicEncrypt(publicKeyPem, Buffer.from("hello", "utf8"));

console.log(ciphertext.toString("base64"));
```

### Node.js 中使用私钥签名

```ts
import { createSign } from "node:crypto";

const sign = createSign("RSA-SHA256");
sign.update("hello world", "utf8");
sign.end();

console.log(sign.sign(privateKeyPem, "base64"));
```

### OpenSSL 中验签

```bash
openssl dgst -sha256 -verify public.pem -signature sign.bin message.txt
```

## 常见问题

### 为什么 RSA 不能直接处理很长的文本

因为 `RSA` 的可处理明文长度会受到密钥位数、填充方式和摘要算法限制。真实项目里更常见的做法是用 `RSA` 保护对称密钥，再用 `AES` 处理真正的大数据内容。

### `RSA-PSS` 和 `RSASSA-PKCS1-v1_5` 应该怎么选

- 新系统优先考虑 `RSA-PSS`
- 旧平台、既有协议或兼容性优先场景更常见 `RSASSA-PKCS1-v1_5`

### 密钥检查适合什么时候使用

当你拿到一段不确定来源的 `PEM` 或 `JWK`，需要快速判断它是公钥还是私钥、位数是多少、当前浏览器能否识别时，密钥检查模式会很有帮助。
